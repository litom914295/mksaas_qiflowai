## 📝 Step 1: 准备测试案例

### 1.1 选择测试用户

从`测试案例数据_v2.1.json`中选择5个典型案例：

- **案例1**：30岁男性，职业困境期（代表年轻人群）
- **案例2**：45岁女性，事业高峰但健康担忧（代表中年成功人群）
- **案例3**：25岁男性，求学/求职压力（代表学生群体）
- **案例4**：55岁女性，准备退休（代表中老年人群）
- **案例5**：35岁男性，多重决策场景（代表决策复杂场景）

### 1.2 生成测试报告

为每个案例生成v2.1完整报告：

```bash
# 运行测试生成脚本（如果已实现）
npm run generate:report -- --case case-001

# 或手动调用报告生成函数
node scripts/generate-test-reports.js
```

### 1.3 分发报告

将生成的报告发送给测试用户（真实用户或内部测试人员）

## 📋 Step 2: 收集用户反馈

### 2.1 发送问卷

向每位测试用户发送：
1. 完整报告（PDF或在线链接）
2. 用户反馈问卷（`用户反馈问卷_v2.1.md`）

### 2.2 填写问卷

用户填写问卷内容：

- **第一部分**：整体满意度（1-5星）
- **第二部分**：v2.1核心功能评价
  - 分级行动清单（可执行性、优先级、影响量化）
  - 决策时间窗口（准确性、置信度、覆盖度）
  - 风水Checklist（实用性、物品推荐、成本）
- **第三部分**：核心指标评估
  - "我知道该干什么"（1-5分）
  - "报告值这个价"（¥168）
  - "我感觉被理解了"（1-5分）
  - "我对未来有信心"（1-5分）
- **第四部分**：v2.0模块回顾
- **第五部分**：开放反馈（喜欢/不满意/建议）
- **第六部分**：测试执行情况（1周后填写）

### 2.3 转换为JSON格式

将问卷答案转换为JSON格式（用于自动分析）：

```json

        "topDislikes": ["无"],
```

## 📊 Step 3: 运行数据分析

### 3.1 运行分析脚本

```bash
cd docs/报告模版/优化方案/v2.2
node analyze-feedback.js feedback-data.json
```

### 3.2 查看分析报告

脚本会自动生成：

1. **feedback-analysis-report.md** - Markdown格式分析报告
2. **feedback-analysis-data.json** - JSON格式数据（供进一步分析）

### 3.3 分析报告内容

报告包含以下部分：

- ✅ **核心指标达标情况**（表格形式）
- 📈 **整体满意度**（平均分、达标率）
- 👥 **用户推荐意愿（NPS）**（推荐者/被动者/贬损者分布）
- 🎯 **行动执行情况**（必做/推荐/加分项完成率）
- 💡 **v2.0模块回顾**（归因分解/希望之光效果）
- ⚠️ **改进建议优先级**（自动识别未达标指标）
- 💬 **用户反馈摘要**（高频词、改进建议汇总）

## 📈 Step 4: 后续跟踪

### 4.1 一周后跟进

1周后联系测试用户，填写问卷第六部分：

- 实际执行的行动项数量
- 实际调整的风水布局数量
- 实际花费时间

### 4.2 一个月后效果反馈

1个月后再次联系用户，收集：

- 主观感受变化（如"心态更积极"）
- 客观指标变化（如"收入提升"、"睡眠改善"）

### 4.3 更新分析报告

将跟踪数据补充到`feedback-data.json`，重新运行分析脚本。

---

## 🚀 Step 5: 基于反馈改进

### 5.1 识别改进优先级

根据分析报告中的"改进建议优先级"部分，确定：

- **高优先级**：达标率<80%的指标，需立即改进
- **中优先级**：达标率80-90%的指标，需适度优化
- **低优先级**：达标率>90%的指标，保持现状

### 5.2 制定改进计划

针对未达标指标，制定具体改进措施：

**示例**：如果"行动清单可执行性"达标率仅75%

- **问题**：用户反馈"行动建议太抽象"
- **改进措施**：
  1. 增加行动步骤的颗粒度（如"晨跑"→"每天6-7点在家附近公园晨跑30分钟"）
  2. 添加更多实例（如"参加行业活动"→"每月参加1次行业meetup，推荐平台：XX"）
  3. 集成现有模板库（action-templates.ts）

### 5.3 迭代测试

改进后，重新进行小规模测试（3-5个用户），验证改进效果。

## 🛠️ 常见问题

### Q1: 如何快速转换问卷为JSON格式？

**A**: 使用以下模板，复制粘贴用户答案：

```javascript
// 问卷转JSON模板
const response = {
  caseId: 'case-XXX',
  caseName: '案例名称',
  userName: '用户姓名',
  feedbackDate: '2025-01-XX',
  responses: {
    overallSatisfaction: 5, // 替换为用户评分（1-5）
    actionExecutability: 5, // 替换为用户评分（1-5）
    timeWindowAccuracy: 4, // 替换为用户评分（1-5）
    fengshuiPracticality: 5, // 替换为用户评分（1-5）
    knowWhatToDo: 5, // 替换为用户评分（1-5）
    understandingFeeling: 5, // 替换为用户评分（1-5）
    hopeFeeling: 5, // 替换为用户评分（1-5）
    nps: 10, // 替换为用户评分（0-10）
    topLikes: ['喜欢1', '喜欢2', '喜欢3'],
    topDislikes: ['不满意1', '不满意2'],

```

### Q2: 如果没有真实用户，可以用内部测试人员吗？

**A**: 可以，但建议：

1. 至少包含1-2个真实外部用户（更客观）
2. 内部测试人员应模拟真实用户角色（不要带开发视角）
3. 如果全部使用内部人员，结果仅供参考（需在生产环境再次验证）

### Q3: 分析报告显示某指标未达标，但样本量很小（如5人），是否可信？

**A**: 样本量小时：

- **达标率90%+**：可信度高，可以进入下一阶段
- **达标率70-90%**：谨慎判断，建议增加样本量或进入A/B测试验证
- **达标率<70%**：严重问题，无论样本量大小都需立即改进

### Q4: 如何处理NPS分数？

**A**: NPS分数解读：

- **NPS≥70**：优秀，用户高度满意
- **NPS 50-70**：良好，有改进空间
- **NPS 30-50**：一般，需重点改进
- **NPS<30**：差，需全面重新设计
